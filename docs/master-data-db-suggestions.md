# ข้อเสนอแนะสำหรับการปรับปรุงฐานข้อมูลข้อมูลหลัก (Master Data DB Suggestions)

จากการเชื่อมต่อ UI เข้ากับฐานข้อมูลจริง (DB) เพื่อจัดการหน้า ข้อมูลหลัก (Master Data) ทั้งหมดในระบบ JBFarmHUB พบว่ามีจุดที่สามารถปรับปรุงเพื่อให้ระบบรองรับ Requirements ที่ซับซ้อนขึ้นในอนาคตได้ดีขึ้น ดังนี้ครับ:

## 1. การรวมตารางที่เกี่ยวข้องกัน (Table Consolidation)
ปัจจุบันมีการแยกตาราง **สาเหตุการตาย (Mortality Causes)** และ **ประเภทการสูญเสีย (Mortality Types)** ออกจากกัน ซึ่งในทางปฏิบัติ ทั้งสองเรื่องนี้มีความเกี่ยวข้องกันอย่างใกล้ชิด
*   **ข้อเสนอแนะ:** อาจพิจารณารวมข้อมูลทั้งสองส่วนนี้เป็นตารางเดียว เช่น ตาราง `LossReasons` โดยมีฟิลด์ระบุประเภท (Type) ว่าเป็นตั๋วการตาย (Death) หรือการคัดทิ้ง (Cull) เพื่อลดความซ้ำซ้อนและทำให้การ Query ง่ายขึ้น
*   **ข้อดี:** ลดจำนวนตารางในฐานข้อมูล, ทำให้ UI ดูแลในหน้าต่างเดียวจบ และลด Join SQL สำหรับ Report

## 2. การเพิ่มฟิลด์ที่จำเป็นในตารางต่างๆ (Field Additions)
จากการ Review โครงสร้างข้อมูลปัจจุบัน ขอเสนอให้จัดการฟิลด์เพิ่มเติมเพื่อให้รองรับ Business Logic ได้ลึกขึ้น:

### ตารางหมวดโรค (Diseases)
*   **เพิ่มส่วนเชื่อมโยงการรักษา:** ควรเพิ่มตารางสอดคล้อง (Mapping table) เช่น `Disease_Treatment_Mapping` เพื่อผูก "กลุ่มโรค" เข้ากับ "มาตรฐานการรักษา" (Treatment Type) หรือ "ยาหลักที่ใช้" (Medication/Item) เพื่อให้เวลาฟาร์มพบโรคนี้ ระบบสามารถ Suggest ยาที่ควรเบิกออกมาใช้ได้ทันที
*   **เพิ่มส่วนระดับความรุนแรง:** เพิ่มฟิลด์ `SeverityLevel` (Low, Medium, High, Critical) ไว้แจ้งเตือนว่าถ้าพบโรคระดับ Critical ต้องเด้ง Alert ไปที่ผู้จัดการฟาร์มทันที

### ตารางประเภทการเลี้ยง (Pig Types)
*   **เพิ่มระยะเวลามาตรฐาน (Standard Days in Phase):** ควรมีฟิลด์ระบุว่าหมูขุน หมูหย่านม จะอยู่ใน Phase นี้กี่วัน เพื่อให้ระบบสามารถคำนวณและประเมิน (Forecast) วันย้ายเล้า หรือ วันจับขายได้อัตโนมัติ
*   **เพิ่มเกณฑ์ของน้ำหนักและ FCR:** เพิ่มฟิลด์ `TargetWeight` และ `TargetFCR` ใช้เป็นเกณฑ์ชี้วัดเปรียบเทียบกับ Performance จริง ณ วันที่ปิด Batch (Batch Closing)

### ตารางสายพันธุ์ (Breeds)
*   **เพิ่มลักษณะเด่นหรือรอบผลผลิต:** ควรเพิ่มฟิลด์ `ExpectedLitterSize` (สำหรับแม่พันธุ์) หรือ `GrowthRate` เพื่อใช้ประเมิน Performance ของพ่อแม่พันธุ์

## 3. โครงสร้างการอ้างอิงของคู่ค้า (Business Partners)
*   **เพิ่มฟิลด์บัญชีและการจ่ายเงิน:** ตารางคูค้าปัจจุบันมีแค่ข้อมูลติดต่อเบื้องต้น (เช่น TaxId, Phone) แต่ในการทำงานกับโมดูลจัดซื้อ (Purchasing) อย่างเต็มรูปแบบ ควรจะมีการระบุ `PaymentTerm` (เครดิตกี่วัน), `Currency` (สกุลเงิน), เพื่อลดกระบวนการการตั้งค่าซ้ำซ้อนตอนสร้างใบ PO
*   **ผูกพันกับกลุ่ม/หมวดหมู่คู่ค้า:** เพิ่ม `PartnerGroupId` เพื่อให้ง่ายต้อการทำ Report และการจำกัดสิทธิ์ (เช่น User นี้เห็นแต่ Supplier หมวดอาหารสัตว์)

## 4. ปรับปรุงเรื่อง Audit Types และ Soft Delete
*   ปัจจุบันบางตารางใช้ `is_active` สำหรับ Soft Delete แล้ว ซึ่งเป็นแนวทางที่ดี แต่หากข้อมูลไหนที่เคยถูกนำไปดึงในเอกสารใบจัดซื้อ/ใบตาย (Transactions) ระบบจะต้องมี Validation เช็คให้แน่ใจว่าไม่สามารถแก้ไขรหัส `Code` ของ Master Data ได้หากว่ามันถูกนำไปใช้แล้ว (ป้องกันข้อมูลเก่าในระบบเปลี่ยนความหมาย)
*   ระบบ Audit Logging `created_date` และ `updated_date` ครอบคลุมแล้ว

---
**สรุปสำหรับสถานะปัจจุบัน:**
ตอนนี้ผมได้ตั้งค่าให้หน้าจอ UI ไป Bind กับ Database ตารางจริงทั้งหมดแล้วโดยยึดตามโครงสร้างของ DB ณ ปัจจุบันครับ
คุณสามารถลองทดสอบระบบว่ามันทำงานและบันทึกข้อมูลเข้าฐานข้อมูลได้จริง และเมื่อถึงช่วง Phase ถัดไป เราสามารถทะยอยนำข้อเสนอแนะเหล่านี้ไปเพิ่มฟิลด์ลงใน DB และตามไปเสริมใน DTO -> UI ต่อไปได้อย่างราบรื่นครับ
